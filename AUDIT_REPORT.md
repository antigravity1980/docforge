## 🔴 Критические ошибки (баги)

### 1. Auth Callback редиректит на `www.docforge.site` (СЛОМАННАЯ АВТОРИЗАЦИЯ через Google)

**Файл:** `src/app/api/auth/callback/route.js`, строка 17

```javascript
const dashboardUrl = 'https://www.docforge.site/dashboard';
```

**Проблема:** После авторизации через Google OAuth пользователь редиректится на `https://www.docforge.site/dashboard`. Но домен `www.docforge.site` настроен на 301-редирект на `docforge.site`. При этом cookie авторизации Supabase привязаны к домену `docforge.site`. Это значит:
- Пользователь попадает на `www` → редирект на основной домен → **cookie теряются** (зависит от настроек cookie, но высокий риск).
- Даже если cookie сохраняются, это лишний редирект, замедляющий UX.

**Рекомендация:** Заменить на `https://docforge.site/dashboard` или использовать динамический `origin`.

### 2. Гонка данных (Race Condition) при обновлении счётчика документов

**Файл:** `src/app/api/generate/route.js`, строки 28–114

```javascript
// Шаг 1: Читаем текущее значение
const currentUsage = profile?.docs_generated_this_month || 0;
// ... генерация документа ...
// Шаг 2: Записываем старое значение + 1
await supabase
    .from('profiles')
    .update({ docs_generated_this_month: (profile?.docs_generated_this_month || 0) + 1 })
    .eq('id', session.user.id);
```

**Проблема:** Если пользователь отправит два запроса одновременно (например, двойной клик или два окна), оба прочитают одно и то же значение `docs_generated_this_month`, и оба запишут `+1`. В результате два документа будут сгенерированы, но счётчик увеличится только на 1. Это позволяет обойти лимиты.

**Рекомендация:** Использовать атомарный SQL-инкремент через Supabase RPC (хранимую процедуру) вместо read-modify-write.

### 3. Несовпадение лимитов между API и Dashboard

**Файл API:** `src/app/api/generate/route.js`, строки 34–38  
**Файл Dashboard:** `src/components/DashboardClient.js`, строка 65

| План | API (`route.js`) | Dashboard (`DashboardClient.js`) |
|------|:-:|:-:|
| Free | 1 | 1 |
| Starter | **30** | **10** |
| Professional | **1000** | **100** |

**Проблема:** Пользователь видит на дашборде лимит `10`, но API разрешает до `30`. Или наоборот — пользователь видит `100`, хотя API даёт `1000`. Это путаница, которая подрывает доверие к продукту.

**Рекомендация:** Вынести лимиты в единый конфигурационный файл (например, `lib/plans.js`) и импортировать его в оба места.

### 4. `getSession()` вместо `getUser()` в серверных API

**Файлы:** `src/app/api/generate/route.js`, `src/app/api/admin/stats/route.js`, `src/app/api/admin/users/route.js`

```javascript
const { data: { session } } = await supabase.auth.getSession();
```

**Проблема:** `getSession()` читает JWT из cookie **без проверки подписи на сервере**. Суть проблемы: злоумышленник может подделать cookie с произвольным JWT, и `getSession()` доверится ему. Согласно [официальной документации Supabase](https://supabase.com/docs/guides/auth/server-side/nextjs), для авторизационных проверок на сервере **нужно использовать `getUser()`**, который валидирует JWT через API Supabase.

**Рекомендация:** Заменить все серверные вызовы `getSession()` на `getUser()`.

## 🟠 Безопасность

### 5. Checkout API не имеет авторизационной проверки

**Файл:** `src/app/api/lemonsqueezy/checkout/route.js`

```javascript
export async function POST(req) {
    // Нет проверки авторизации!
    const { variantId, userEmail, userId, planName } = await req.json();
}
```

**Проблема:** Любой может вызвать этот API с произвольным `userId` и `userEmail`, и создать чекаут-сессию от имени другого пользователя. Хотя сама оплата защищена LemonSqueezy, это позволяет:
- Спамить созданием чекаут-ссылок.
- Потенциально привязать оплату к чужому аккаунту.

**Рекомендация:** Проверять аутентификацию и убедиться, что `userId` совпадает с авторизованным пользователем.

---

### 6. SQL-инъекция через параметр поиска в Admin Users API

**Файл:** `src/app/api/admin/users/route.js`, строка 31

```javascript
query = query.or(`email.ilike.%${search}%,full_name.ilike.%${search}%`);
```

**Проблема:** Переменная `search` подставляется напрямую в строку фильтра PostgREST. Хотя Supabase Client Library экранирует часть символов, использование шаблонных строк с пользовательским вводом в `.or()` — плохая практика. Специальные символы PostgREST (`,`, `.`, `(`, `)`) могут исказить запрос.

**Рекомендация:** Использовать отдельные `.ilike()` фильтры вместо строковой подстановки.

---

### 7. Webhook LemonSqueezy не имеет идемпотентности

**Файл:** `src/app/api/lemonsqueezy/webhook/route.js`

**Проблема:** Если LemonSqueezy отправит один и тот же вебхук повторно (retry), план пользователя будет обновлён повторно. Это не критично сейчас, но может вызвать проблемы в будущем, если добавится логика начисления бонусов или отправки email-уведомлений.

**Рекомендация:** Сохранять `webhook_event_id` в БД и проверять дубликаты.

### 8. Список админских email-адресов захардкожен в коде

**Файл:** `src/lib/config.js`

```javascript
export const ADMIN_EMAILS = ['artpavelalex@gmail.com', 'antigravity1980@gmail.com'];
```

**Проблема:** Реальные email-адреса находятся в публичном репозитории GitHub. Это:
- Раскрытие персональных данных.
- Невозможность добавить/удалить админа без деплоя.

**Рекомендация:** Вынести в переменные окружения (`ADMIN_EMAILS`) или в таблицу `settings` в БД (где уже хранятся другие настройки).

## 🟡 Производительность и узкие места

### 9. Middleware выполняет запрос к базе данных на КАЖДЫЙ запрос

**Файл:** `src/utils/supabase/middleware.js`, строки 90–128

```javascript
// 4. Maintenance Mode Check
const { data: setting } = await supabase
    .from('settings')
    .select('value')
    .eq('key', 'maintenanceMode')
    .single();
```

**Проблема:** Каждый HTTP-запрос (включая загрузку CSS, шрифтов, favicon) выполняет запрос к базе данных Supabase для проверки режима обслуживания. При нагрузке в 1000 запросов/мин это ~1000 лишних запросов к БД в минуту.

**Рекомендация:**
1. Кэшировать значение `maintenanceMode` на уровне Edge (например, через `unstable_cache` или KV).
2. Или проверять только для HTML-страниц, а не для всех ресурсов (сейчас `pathname.includes('.')` частично фильтрует, но не надёжно).

### 10. AI-генерация не имеет таймаута и retry-логики

**Файл:** `src/lib/ai.js`

```javascript
const response = await fetch(GROQ_API_URL, { ... });
```

**Проблема:** Если AI-провайдер (Groq, DeepSeek, OpenRouter) зависнет, запрос будет ждать бесконечно. Vercel имеет свой таймаут для Serverless Functions (по умолчанию 10 секунд на Hobby, 60 секунд на Pro), но пользователь не получит понятного сообщения об ошибке.

**Рекомендация:**
1. Использовать `AbortController` с таймаутом 30 секунд.
2. Показывать пользователю информативное сообщение при превышении таймаута.

---

### 11. Конфигурация AI провайдера читается из БД при КАЖДОЙ генерации

**Файл:** `src/lib/ai.js`, функция `getAIConfig()`

```javascript
export async function generateWithAI(systemPrompt, userPrompt) {
    const config = await getAIConfig(); // запрос к БД каждый раз
}
```

**Проблема:** Каждый вызов генерации делает отдельный запрос к таблице `settings` для получения провайдера и модели. Эти настройки меняются крайне редко.

**Рекомендация:** Кэшировать конфигурацию на уровне модуля с TTL 60 секунд.

---

### 12. DeepSeek игнорирует настройку модели

**Файл:** `src/lib/ai.js`, строка 130

```javascript
async function callDeepSeek(systemPrompt, userPrompt, model) {
    // ...
    model: 'deepseek-chat', // параметр model игнорируется!
}
```

**Проблема:** Функция принимает параметр `model`, но всегда использует захардкоженное значение `'deepseek-chat'`. Если администратор выберет другую модель DeepSeek в настройках, она будет проигнорирована.

**Рекомендация:** Использовать `model || 'deepseek-chat'`.

## 🟣 Архитектура и качество кода

### 16. Admin Layout создаёт вложенный `<html>` тег

**Файл:** `src/app/admin/layout.js`, строки 29–104

```jsx
return (
    <html lang="en">
        <body>
            ...
        </body>
    </html>
);
```

**Проблема:** Корневой `layout.js` (`src/app/layout.js`) уже генерирует `<html>` и `<body>`. Повторное создание этих тегов в admin layout приводит к **невалидному HTML** с вложенными `<html>` элементами. Браузеры обрабатывают это по-разному, но это может вызвать:
- Сломанные стили.
- Проблемы с гидратацией React.
- SEO-штрафы.

**Рекомендация:** Удалить `<html>` и `<body>` из admin layout, оставив только `<div>` обёртку.

---

### 17. Два разных Supabase-клиента на клиенте

**Файлы:**
- `src/utils/supabase/client.js` — фабрика `createClient()`, создаёт новый экземпляр при каждом вызове.
- `src/lib/supabase.js` — экспортирует синглтон `export const supabase = createClient()`.

**Проблема:** Разные компоненты используют разные способы:
- `Header.js` импортирует `import { supabase } from '@/lib/supabase'` (синглтон).
- `DashboardClient.js` вызывает `const supabase = createClient()` внутри компонента.
- `SignInClient.js` вызывает `const supabase = createClient()` внутри компонента.

Это приводит к тому, что `Header` может иметь устаревшую сессию (если синглтон был создан до авторизации), а другие компоненты — нет. Потенциальный источник багов вида «неправильно отображается статус авторизации».

**Рекомендация:** Убрать `src/lib/supabase.js` и везде использовать `createClient()` из `src/utils/supabase/client.js`.
